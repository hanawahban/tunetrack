# AGENTS.md

Source of truth for any agent (or human) working in this repo. `CLAUDE.md` just points here — keep this file current instead of maintaining two.

## What this is

TuneTrack — a Last.fm-style scrobbling app styled as a used-record shop. See `PRODUCT.md` for the product vision and `DESIGN.md` for the visual design system. This file covers layout, running it, and code conventions.

## Layout

```
api/                  ElysiaJS + Drizzle API, Bun runtime
  src/
    <resource>/        one module per resource: albums, artists, tracks, users, auth, scrobbles, stats
      model.ts          TypeBox request/response schemas
      service.ts        DB access via Drizzle — abstract class + static methods (no HTTP/context dependency)
      index.ts          routing only — no db.* calls, no inline t.Object() schemas
      *.test.ts / *.integration.test.ts   co-located tests (bun:test)
    common/
      models.ts          shared named Elysia models (.model()) so @elysiajs/swagger emits real
                          components/schemas entries, letting Orval generate proper named DTOs
      pagination.ts       cursor pagination: encode/decode cursor, resolveLimit, paginatedResponse
      http-error.ts        Nest-style {statusCode, message, error} response builder
      pg-error.ts          unwraps DrizzleQueryError to the real Postgres SQLSTATE code
    auth/
      guard.ts             JWT plugin + `auth`/`roles` macros (role-based route guards)
      current-user.ts       safe-user select, toUserResponse, shared role schema
      routes.ts              /auth/register, /auth/login
    db/
      schema.ts             Drizzle schema — source of truth for the DB shape
      index.ts                Drizzle client (bun-sql driver)
      seed.ts                  creates the 3 test accounts (see below)
    test/
      app.ts                 composed Elysia app (no swagger/cors/listen) + request helpers for tests
      db.ts                   pglite-backed test DB + resetDb()
      preload.ts               bun:test preload — mocks '../db' to the pglite instance
    feature/                end-to-end flow tests
  drizzle/
    *.sql                   migrations, generated by drizzle-kit, never hand-edited
    meta/                    drizzle-kit's migration journal
  docs/openapi.json         generated OpenAPI spec (API writes it on boot) — committed, regenerated
                             whenever the API changes
web/                   React 19 + Vite
  src/
    pages/              route-level components
    components/
      ui/                shadcn primitives — migrating Radix → Base UI (see #17, in progress)
      records/            feature components (album/track dialogs, vinyl sleeve, etc.)
      layout/              shell/nav
    lib/
      api/generated/       Orval-generated TanStack Query client from ../api/docs/openapi.json —
                            gitignored, regenerated via predev/prebuild or `pnpm --filter web run orval`
      api-types.ts          thin aliases for the handful of DTOs Orval can't name directly
                            (Elysia/TypeBox limitation — see #30/#35/#40/#41)
      auth-context.tsx      session state from decoded JWT (display only, not authorization)
    test/                  vitest setup
pnpm-workspace.yaml    lists api + web
package.json           workspace root — shared scripts only
```

## Running it

Full walkthrough with troubleshooting is in `README.md`. Condensed:

```bash
pnpm install                          # installs api + web from the workspace root

# api
cp api/.env.example api/.env          # set DATABASE_URL, JWT_SECRET
pnpm --filter api run db:migrate
pnpm --filter api run db:seed
pnpm --filter api run start:dev       # :3000, Swagger at /api/docs

# web (separate terminal)
cp web/.env.example web/.env
pnpm --filter web run dev             # :5173
```

Test accounts (password `password123` for all): `admin@test.com`, `curator@test.com`, `listener@test.com`.

Correctness gates:
```bash
pnpm --filter api run typecheck   # tsc --noEmit
pnpm --filter api run test        # bun:test — unit/integration/feature suite against pglite
pnpm --filter web run build       # tsc -b && vite build
pnpm --filter web run lint        # oxlint
pnpm --filter web run test        # vitest
```

## Conventions

- **Package manager**: pnpm workspace only. Root `pnpm-workspace.yaml` lists `api` and `web`; one `pnpm install` at the root produces a single `pnpm-lock.yaml` — no `package-lock.json`, no `bun.lock`, no per-package lockfiles.
- **Runtime**: `api` runs on Bun (not Node) — `bun src/index.ts`, Bun's native `Bun.password` and `bun-sql` driver. `web` builds/dev-serves via Vite as usual. pnpm here is the workspace/package-manager layer only — it doesn't change how either package actually runs.
- **Model/service/controller split** (per resource): `model.ts` owns TypeBox schemas only, `service.ts` owns DB access + serialization (abstract class, static methods), `index.ts` owns routing only.
- **Shared named models**: response schemas reused across modules (`AlbumResponse`, `TrackResponse`, `Role`, etc.) are registered via Elysia's `.model()` in `common/models.ts`, so `@elysiajs/swagger` emits them as real `components/schemas` entries instead of duplicating an anonymous inline schema at every usage site — Orval then generates proper named DTOs on the frontend instead of per-endpoint anonymous types.
- **Auth pattern**: `auth/guard.ts`'s `authGuard` exposes `auth: true` (any authenticated role) and `roles: [Role, ...]` macros, applied per-route. Re-fetches the current user's role from the DB on every request rather than trusting the JWT payload, so a role change takes effect on the requester's very next request without re-login.
- **3-role RBAC**: `LISTENER` (browse + scrobble only), `CURATOR` (+ full CRUD on artists/albums/tracks), `ADMIN` (+ `PATCH /users/:id/role`). An admin can't change their own role or another admin's role (403), regardless of the requested role.
- **Authorization is server-side only.** The frontend decodes the JWT client-side (`lib/jwt.ts`) purely to display role/gate UI controls — never treat that as the actual authorization boundary; the guards are.
- **Cursor pagination**: `artists`/`albums`/`scrobbles` list endpoints return `{items, nextCursor}`, sorted on a compound unique key (`createdAt, id` or `playedAt, id`) descending. `nextCursor` is an opaque base64url-encoded pair; a malformed cursor degrades to "start from the beginning" rather than erroring. `common/pagination.ts` clamps `limit` into `[1, MAX_PAGE_SIZE]`.
- **Services signal domain outcomes, not raw DB errors.** FK violations on create/update map to a 404 (`pg-error.ts` unwraps Drizzle's wrapped Postgres error to the real SQLSTATE code), unique violations to 409, rather than letting a raw driver error surface as an unhandled 500.
- **Testing**: `bun:test` against `@electric-sql/pglite` (in-memory Postgres, migrated fresh per test run via `test/db.ts`) — never the shared dev DB. `test/app.ts` composes the route modules without swagger/cors/listen for in-process request testing, plus `get`/`post`/`patch`/`del` request helpers and `registerAs()`/`login()` auth helpers. Frontend uses `vitest` + Testing Library.
- **Env sync**: `api`'s `WEB_ORIGIN` and `web`'s `VITE_API_URL` must point at each other's real address/port or CORS breaks. If Vite bumps off 5173 (port taken), either free it or update `WEB_ORIGIN` and restart the API.
- **Commits**: Conventional Commits (`feat(scope): ...`, `fix(scope): ...`, `docs: ...`). No AI-attribution footers in commits, PR titles/bodies, or comments.
- **Branching**: one feature branch per issue, named for the issue/ticket where one exists.
